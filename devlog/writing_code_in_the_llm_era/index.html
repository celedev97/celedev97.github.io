Writing Code in the LLM Era, are we still programmers?

2026-05-16 · #llm · #ai · #coding

I started writing code in an old era, where stack overflow didn’t exist yet, coding was seen as a dark art, and you had to read books to learn how to do it.
It was a time where you had to write code without any help, and you had to figure out everything by yourself, the only small help you could get was from online forums, but they were not very popular, and you had to wait for hours or even days to get an answer to your question, and often the answer was not very helpful.

Coding was a slow art that required a lot of patience, write, compile, run, debug, repeat, and if you were lucky you could get a working code after a few hours of work.
It was a gruelsome process, but it was also a very rewarding one, because you had to put a lot of effort to get something working, and when you finally got it working, it was a great feeling of accomplishment.

In the last few years thanks to my university student status i had the chance to try out Github Copilot, and it was a game changer, at first it was only a code completion tool, but it was already a huge help, code completion was not just about completing the current word without failing a variable name, it was about completing the whole line of code, and even the whole function, it was like having a coding assistant that could help you write code faster and more efficiently.
And the scariest part? It was actually good, it was not just a random code generator, it was getting context from the code you were writing, and it was able to generate code that you would have written yourself, but faster, it was like having a super fast pair programmer, sometimes kinda dumb but often good enough and blazingly fast.

At first I was a bit skeptical about using it, I thought it would make me lazy, and that I would lose my coding skills, but I quickly realized that it was not the case, it was actually helping me to write better code, because it was giving me suggestions that I would not have thought of, and it was also helping me to learn new things, because it was generating code that I had never seen before, and it was also helping me to write code faster, because I could read, review, understand and accept the suggestions much faster than I could write the code myself.

Then the big change came… Github Copilot implemented a chat inside the IDE, i kind of hated it at first because i never liked the idea of having a chat inside my code editor, and i never liked the idea of talking to a bot, I am the human, I do the thinking and I do the coding, every tool that can make it faster is welcome, but a chat?
I didn’t see the point of it, I thought it would be just a gimmick, but time proved me wrong, once the chat stopped being just a chat and started becoming an agent the whole experience changed, it was not just about asking questions and getting answers, it was about giving it tasks and letting it do the work for you.

It was like having a personal coding assistant that could do the boring and repetitive tasks for you, that boring CRUD you’ve written a hundred times? “Write me a CRUD based on how is it done in this other class”, and it would generate the code for you.

Test driven development? A great idea, but a lot of work, writing tests is often seen as a chore, i like creating stuff, writing the tests is not creating, it’s just writing code that checks if the code you created is working, it’s not fun, but it’s necessary, and now? “Write me tests for this class”, and the tests are done.

The LLM era is here, and it’s changing the way we write code, it’s making it faster, more efficient, and more fun.
But it is really just this? Do we still write code? Do we still need to know how to code? What is the purpose of learning how to code if you can just ask an LLM to do it for you?

I think the answer is “kinda”, we still need to know how to code, because we need to understand what the LLM is doing, we need to be able to read and understand the code that it generates, and for that we need to have a great understanding of coding concepts, of frameworks, of libraries, of design patterns, and of best practices, we need to be able to review the code that it generates and make sure that it’s good, that it’s secure, that it’s efficient, and that it’s maintainable.

But how about just opening the IDE with an idea and letting your fingers fly on the keyboard? What about writing code without having a clear idea, whatching the code unfold in front of you while you are writing it, whatching the shape of the repository take form while you go forward, what about the joy of writing code, of creating something from nothing, of solving a problem with code?
What about the joy of learning new things of creating new things, of exploring new technologies, of pushing the boundaries of what is possible with code?

Now that an LLM can do all of that for you, what is left for us to do? Are we just going to be the ones who review the code that the LLM generates? Are we just going to be the ones who give it tasks and let it do the work for us? Are we just going to be the ones who ask it questions and get answers from it?

I don’t have the answer to those questions, but I think that the LLM era is not just about writing code faster, it’s about changing the way we think about code, code is not the final product anymore, it’s just a means to an end, and the way it’s created is not the most important thing anymore.
As long as the code is clean, efficient, secure, and maintainable, does it really matter if it was written by a human or by an LLM?

Two years ago i would have said “yes it matters, code written by a machine will never have the soul of the code written by a human”, but now i am not so sure anymore.
Maybe the soul of the code is not in the code itself, but in the idea behind it, in the problem that it solves, in the impact that it has on the world, maybe the soul of the code is not in the code itself, but in the person who created it, in their vision, in their creativity, in their passion for coding.

But how can someone still love coding if all it takes to do it is describe the final product to a machine?
I think that the love for coding is not just about writing code, it’s about creating something from nothing, it’s about solving problems with code, it’s about learning new things, it’s about exploring new technologies, it’s about pushing the boundaries of what is possible with code, and as long as we can do all of that, I think that we can still love coding, even if we are not the ones writing the code anymore.

Maybe the future of programmers is not about being the fastest on the keyboard and the sharpest in knowing a language’s caveats, but about being a good software architect, a good problem solver, a good designer, a good reviewer, a good tester, a good maintainer, and maybe for some even a good communicator.

I don’t know what the future holds for programmers, i do sometime miss the old days of coding, but then i realize that i rewrote my whole website in a new framework in less than 30 minutes and i think “would i really have done that without an LLM? Or would i have just kept the old code and never had the motivation to rewrite it?”

The future is uncertain, the future is here, the future is fast, it’s changing and we don’t know where it’s going, but one thing is for sure: the love of a programmer for creating new things, for solving problems, for learning new things, for exploring new technologies, will never die, and if we can do all of that faster?
Then why not?
Then why not rewrite that old code that you never had the motivation to rewrite? Then why not create that new project that you always wanted to create but never had the time to create? Then why not learn that new technology that you always wanted to learn but never had the time to learn? Then why not push the boundaries of what is possible with code?

Writing code has changed, it has become faster, less personal, but that doesn’t mean that it needs to be less enjoyable, less creative, less rewarding, it just means that we need to change the way we think about code, we need to change the way we think about programming, and we need to change the way we think about ourselves as programmers.

We cannot be just code monkeys anymore, we need to be more than that, we need to be creators, architects, project managers, orchestrators of a swarm of agents.
Was it was i wanted to be when i started coding? No.
Will it be fun? I don’t know, but i have a positive feeling and i am excited to see where this new era of programming will take us.
