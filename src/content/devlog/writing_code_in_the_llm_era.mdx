I started writing code in an old era — before Stack Overflow existed, when coding was seen as a dark art and you learned from books.<br/>
If you got stuck, you posted on a forum and waited hours or days for an answer.<br/>
Often the answer wasn't even helpful.

Coding was slow. Write, compile, run, debug, repeat. Grueling — but also deeply rewarding.<br/>
When something finally worked after hours of fighting, the sense of accomplishment was real.

## Then the Copilot came in...

Thanks to my university student status I got early access to GitHub Copilot, and it was a game changer.

It was not just autocomplete like the intellisense or stuff like that — it completed entire lines, entire functions, pulling context from the surrounding code.<br/>
It felt like having a blazingly fast pair programmer. Sometimes dumb, often good enough, but always extremely fast.

At first I was skeptical. I thought it would make me lazy, erode my skills.<br/>
It didn't. It actually made me write *better* code — surfacing approaches I wouldn't have thought of, generating patterns I hadn't seen before, and thanks to my curiosity it even becaming a way of learning *Why did it write it like this? I need to understand*<br/>
Reading, reviewing, and accepting a suggestion is much faster than typing from scratch, and you still understand every line.

## The agent shift

Then came the chat inside the IDE, and I hated it immediately.

*I am the human. I do the thinking. I do the coding.*<br/>
A chat felt like a gimmick.

Oh boy I was very much fucking wrong....

Once the chat stopped being a chat and started becoming an agent, everything changed.<br/>
It wasn't about asking questions — it was about delegating tasks.

That boring CRUD you've written a hundred times?

> *"Write me a CRUD based on how it's done in this other class."*

Done.

Test driven design? Great idea for avoiding regression, boring to mantain, a terrible chore.<br/>
I like creating things, writing code that checks if the code works is not creating, it's just boring...

And now?

> *"Write me tests for this class."*

Done.

## Then came Claude

Copilot was impressive. Claude is a different category.

It was not about just smarter suggestions or delegating small tasks, it is a full agent that reads your entire codebase, plans changes across multiple files, writes code, runs commands, reads the output, fixes its own mistakes, and iterates.<br/>
You describe a goal. It executes.

I rewrote this entire website from Hugo to Astro — new framework, new theme, custom terminal design, animated typewriter effect, MDX support — and I barely typed a line of code myself.<br/>
Directed it? Yes.<br/>
Reviewed it? Yes.<br/>
Wrote it? Not really.<br/>

And here's the uncomfortable part: it did it faster and with fewer mistakes than I would have alone.

The gap between what a senior developer can produce in a day and what Claude can produce in an hour is no longer a gap, it's a canyon.<br/>
The average programmer, grinding through boilerplate, looking up docs, copy-pasting from Stack Overflow... that workflow is effectively obsolete.

This doesn't mean programmers are obsolete.<br/>
But it does mean that *being average is no longer enough*.<br/>
If you bring the same value to a task that an LLM does, you are being outcompeted.<br/>
Silently, cheaply, and at scale, because 20$ of Claude Pro can probably do it faster than what a programmer can.

## Are we still programmers?

The LLM era is making coding faster, more efficient... but it raises an uncomfortable question:

*If a machine can write the code, do we still need to know how to code?*

I think the answer is **yes, but differently**.

You still need to understand what the LLM is doing. <br/>
You need to read and evaluate the code it generates, spot security issues, performance problems, design mistakes.<br/>
For that, you need a solid grasp of frameworks, patterns, best practices, and trade-offs.<br/>
The LLM is only as useful as the person directing it.

## What about the joy of it?

Here's the harder question.

What about opening the IDE with a half-formed idea, watching the code take shape as you write it, whatching the repository growing, the architecture take shape while you go forward and reorganize your code because it need to be clean, it need to be perfect... ?<br/>
What about the pure satisfaction of solving a problem with your own hands?

When an LLM can replicate that process in minutes, what's left?

I don't have a clean answer.<br/>
But I don't think the soul of code lives in the keystrokes.<br/>
It lives in the idea, in the problem being solved, in the impact it has.<br/>
The creativity, the vision, the obsession with getting it right... those are still ours.

## Where this leaves us

Two years ago I would have said *code written by a machine will never have the soul of code written by a human.*<br/>
Now? I'm not so sure anymore, often Claude can write stuff in a better way than the average programmer if directed correctly.

The role is shifting.<br/>
The programmer of the future is less about being the fastest typist or knowing every language quirk, and more about being:

- a good architect
- a sharp problem solver
- a rigorous reviewer
- a clear communicator

Was that what I imagined when I started coding? No.<br/>
Will it be fun? I genuinely don't know — but I'm curious.

## Closing thought

I sometimes miss the old days...<br/>
I sometime fear that by keeping giving directions and reviewing code i will lose my *magic touch*...<br/>
Then I remember that I rewrote my entire website in a new framework in under 30 minutes.

Would I have done that without an LLM? Probably not.<br/>
The old version would have stayed forever, rotting quietly, it was not a priority.

So instead of thinking about the lost joy of hitting the keyboard and hoping the codes compiles...<br/>
Why not rewrite that old codebase you've been avoiding?<br/>
Why not finally start that side project you've been putting off for years?<br/>
Why not learn that technology you never had time for?

Writing code has changed, it's faster, less personal.<br/>
But it doesn't mean that it needs to be less creative or less rewarding.<br/>
It just means thinking of ourselves differently: not as code monkeys, but as orchestrators of an army of mechanical code monkeys.

The love of building things, solving problems, and exploring what's possible?<br/>
That part didn't go away.<br/>
It just got faster, and while that may be weird at first... it means we can achieve so much more, create so much more, and if that's the case...<br/>
Is it worth it to lose time fighting nostalgia instead of jumping on the train of creativity and let the LLM do the heavy lifting?
